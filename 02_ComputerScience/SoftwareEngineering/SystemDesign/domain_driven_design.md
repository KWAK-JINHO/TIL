# 마이크로서비스 프리미엄

## 안티패턴

- 마이크로서비스 아키텍처의 관심사 중 하나는 도입하기 위해 어떤 기준으로 서비스를 나눌 것인가?
- 자동화된 배포, 모니터링, 실패처리, 최종 일관성 등 분산 시스템의 모든 것
- 성급한 분리는 서비스간 결합도가 높아진다.

MSA를 공부하다가 보면 DDD를 마주하게 된다.
물리환경을 분리하면 각서비스간의 통신이 잦아지고 결합도가 높아져 난감하다.

## 해결책

- MSA가 더 나은것같아도 일단 모놀리식으로 시작
- YAGNI(you aren't gonna need it)
- 소리치는 아키텍쳐(소프트웨어의 아키텍쳐만 보아도 어떤 유스케이스를 가지고 있는지 명확해야한다.)도 잠시 미뤄둔다
- 먼저 올바른 바운디드 컨텍스트를 찾는다.(복잡한 시스템에서 같은 단어(예:고객)라도 컨텍스트마다(예:회원관리, 결제, 고객지원) 주요 속성과 기능이 다르기 때문에 컨텍스트를 어떻게 나눌지 경계선을 정하는 과정)
- 단일패키지 -> 패키지 -> 모듈 -> MSA 순으로 진행

# 완벽한 설계

## 안티패턴

- 완벽한 설계란 없다.
- 요구 사항은 끊임없이 변하고 기획자도 이를 파악하지 못한다.

## 해결책

- 변화에 맞서지 말고 변화 자체를 내재화하는 프로세스를 만들어라
- 일단 구현해라. 그를 통해 충분한 도메인 지식을 얻기 위해 노력한다.

변화에 맞춰 진행하는걸 '애자일' 이라한다.
애자일 + 객체지향프로그래밍 = DDD (그렇다고 OOP가 DDD 와 동일하지는 않다.)

- getter와 setter를 지양한다.
  처음에는 써도 이를 점점 리펙토링해나가면서 풍부한 도메인 모델로 바꿔나간다.

# 유비쿼터스 언어

## 용어 사전

- 클래스, 함수 이름을 짓기가 너무 어렵다.
  유비쿼터스 언어(언제 어디서나 존재하는 흔한 언어)를 도입

- 바운디드 컨텍스트가 없다면, 바운디드 컨텍스트를 꼭 먼저 정의해서 그에 맞게 지을 필요 없다.
- 한글 이름, 영문이름 모두 정해서 용어 사전에 기록
- 명사뿐 아니라 동사도 대상

- 살아있는 문서로 만들기 위해 프로젝트에 README.md 파일로 유지(코드 형상관리와 함께)
- 용어도 리뷰 대상이며 더 의미 있는 용어를 지속적으로 찾아야 한다.
- 용어에 신경쓰지 않으면 언젠가 후의 개발자가 해당 코드의 의미를 해석해야 하는 부담을 떠안게 된다.
  예시)

## 인수 테스트

인수 조건: 특정 기능이 완료 되었다가 판단할 수 있는 명확하고 측정 가능한 요구사항.
이를 기반으로 테스트코드를 만들면 자연스럽게 용어에 대해 이야기하게 된다.

예시)
'과제' 라는 단어는 교수님 입장에서는 풀라고 내어주는것이고 학생입장에서는 제출해야하는 것 이기 때문에 내어주는건 '과제' 학생들이 제출하는건 '과제 제출물' 로 용어를 정의

# 도메인 지식 탐구

## 비즈니스 도메인

- 우리는 어떤 종류의 비즈니스를 하고 있는가?
- 선두 업체, 경쟁 업체는 어떤 기능을 제공하는가?

어느정도까지 기능의 확장을 열어두어야 할까(OCP)짐작할 수 있다.

## 하위 도메인

- 전자 상거래에는 상품, 전시, 주문, 결제, 정산 등의 하위 도메인이 있다.
- 외부 시스템을 사용할 때도 있다.
- 다른 서비스와 차별화 되는 점
- 자원을 어디에 집중할 것인가
